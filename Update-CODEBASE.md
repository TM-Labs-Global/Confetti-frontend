# Confette - Codebase Structure

> **Stack:** Next.js 15 (App Router) · React 19 · TypeScript · Tailwind CSS v4 · Prisma 5 (SQLite) · jose (JWT) · lucide-react

---

## Overview

Confette is an **AI-assisted event planning and vendor marketplace** platform built for the Nigerian market. It is a **full-stack Next.js application** - the frontend (React pages and components) and the backend (API routes, database, auth) all live in this single repository, written entirely in **TypeScript**.

There are three distinct user roles, each with its own isolated portal:

| Role | Portal path | Responsibility |
|---|---|---|
| **Organiser** | `/organiser` | Creates event plans, allocates budgets, reviews vendor bids |
| **Vendor** | `/vendor` | Browses the marketplace, submits bids on plan categories |
| **Admin** | `/admin` | Oversees all plans, escrow accounts, and platform activity |

### Data Models

```
User
  ├── id, name, email, password (bcrypt-hashed)
  ├── role: "organiser" | "vendor" | "admin"
  ├── plans[]         → Plan
  ├── bids[]          → Bid
  └── notifications[] → Notification

EventType
  ├── id, name, description
  ├── categories[]    → Category
  └── plans[]         → Plan

Category
  ├── id, name, description, defaultPct (budget allocation %)
  └── eventTypeId     → EventType

Plan
  ├── id, name, status ("draft" | "open" | "closed")
  ├── date, dateFlexible, state, city, totalBudget
  ├── shareCode (unique 7-character shareable token)
  ├── organiserId     → User
  ├── eventTypeId     → EventType
  ├── categories[]    → PlanCategory
  └── bids[]          → Bid

PlanCategory
  ├── id, name, allocation (₦ amount)
  ├── planId          → Plan
  └── bids[]          → Bid

Bid
  ├── id, amount, pitch, status ("pending" | "accepted" | "rejected")
  ├── isCounterBid, counterReason, canUpdate
  ├── planId          → Plan
  ├── planCategoryId  → PlanCategory
  └── vendorId        → User

Notification
  ├── id, type, message, isRead
  └── userId          → User
```

Prisma auto-generates TypeScript types from `schema.prisma`. Import them directly:

```ts
import type { Plan, Bid, User, PlanStatus } from "@prisma/client";
```

#### Portal Layouts - Auth Guard Pattern

Each portal layout (`organiser/layout.tsx`, `vendor/layout.tsx`, `admin/layout.tsx`) follows the same TypeScript pattern:

```tsx
// Example: src/app/organiser/layout.tsx
"use client";

import { useAuth } from "@/features/auth/context/AuthContext";
import { OrgSidebar } from "@/features/organiser/components/OrgSidebar";
import { TopBar } from "@/features/organiser/components/TopBar";
import { redirect } from "next/navigation";

export default function OrganiserLayout({ children }: { children: React.ReactNode }) {
  const { user, loading } = useAuth();

  if (loading) return <LoadingScreen />;
  if (!user) redirect("/login");
  if (user.role !== "organiser") redirect(`/${user.role}/dashboard`);

  return (
    <div className="shell">
      <OrgSidebar />
      <main>
        <TopBar />
        {children}
      </main>
    </div>
  );
}
```

## Authentication Flow

Unchanged in behaviour; migrated to TypeScript.

```
Browser                          Next.js Server
  │                                    │
  │  POST /api/auth/login              │
  │ ─────────────────────────────────► │  1. Verify email + bcrypt password
  │                                    │  2. signToken({ id, role, name, email })
  │  Set-Cookie: confette_token=...    │  3. Return AuthUser + HttpOnly cookie
  │ ◄───────────────────────────────── │
  │                                    │
  │  (subsequent requests)             │
  │  Cookie: confette_token=...        │
  │ ─────────────────────────────────► │  4. getAuthUser() reads + verifies JWT
  │                                    │  5. API handler proceeds with typed user
```

- Tokens are **HttpOnly** (no JS access), SameSite=Lax, 7-day expiry
- `AuthContext` calls `GET /api/auth/me` on mount to restore session state
- Portal layouts enforce role access **client-side**; all API routes enforce it **server-side**
